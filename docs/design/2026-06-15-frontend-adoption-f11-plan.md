# Plan: Adopt new React frontend + full F11 (wallet provenance attestation)

> Source: brainstorming + plan-mode session 2026-06-15. Implements the new `Cortexfrontend`
> UI and feature F11 (`docs/superpowers/specs/2026-06-15-provenance-attestation-design.md`,
> PRD F11, TASKS F11.1–F11.8).

## Context

Cortex's public UI is currently an **Eleventy v3 static site** (`site/`) that fetches on-chain
data at build time via `sui`/`walrus` CLIs and deploys to Walrus Sites from `site/dist/`. The user
has a new, polished frontend in `github.com/ahmadsyafiqkamil/Cortexfrontend` and wants it to become
the production Cortex UI, while also executing the **F11 provenance-attestation** feature.

The new repo is a **Vite 6 + React 18 + react-router 7 + Tailwind 4 + shadcn/ui** SPA. It ships
**two designs**; per user decision we use the **indigo / Figma-spec** one:
`src/app/data/mock.ts` + `src/app/screens/{HomeScreen,PageDetailScreen,GraphScreen}.tsx` +
`src/app/components/{NavBar,PageCard,TagPill,ConfidenceBadge,MonoAddress}.tsx`. These are **fully
built but not wired** — `routes.tsx` currently mounts the *other* (brutalist) `pages/` design under
`/app/...`. The indigo `NavBar`/screens expect routes `/`, `/:slug`, `/graph`, `/sources`.

User decisions (locked):
1. **Design** = indigo Figma-spec (`screens/` + `data/mock.ts`).
2. **Data** = build-time JSON injection (port `site/src/_data/*.js`; stays static; Walrus-Sites-friendly).
3. **F11** = full, including the Move `cortex::attest` module + package upgrade + `cortex attest` CLI
   + native `@mysten/dapp-kit` Verify panel.

Outcome: a new indigo React SPA, served from `site/dist/` on Walrus Sites, rendering **real**
Cortex wiki data, with a working wallet "Attest Provenance Verified" flow writing on-chain.

Today is **2026-06-15**; submission **20–21 Jun**. Keep scope disciplined (CLAUDE.md rule #6).

## Phase 2 — Build-time real-data injection (reuse `_data/*.js`)

Create **`site/scripts/fetch-cortex-data.mjs`** that ports the existing fetchers and writes
`src/app/data/cortex-data.json` matching the `data/mock.ts` types (`Source`, `Page`, `Claim`,
`Version`, `Dispute`). Wire it as a prebuild: `"prebuild": "node scripts/fetch-cortex-data.mjs"`
(so `pnpm build` runs it first). `data/mock.ts` becomes a thin loader that imports the JSON (keep
the exported types + `pageBySlug`/`sourceById` helpers).

Reuse logic from (read these, port the helpers):
- `site/src/_data/config.js` → packageId, wikiId, network, explorer URLs (from
  `agent/.cortex/config.json`).
- `site/src/_data/pages.js` → `suiJSONSafe`, `readBlob` (cache-first then
  `walrus read <blob> --context testnet`), `parseFrontmatter`, `extractMarkers` (`^[blob:ID]`),
  `extractWikilinks` (`[[slug|disp]]`), per-claim confidence. Dynamic-field scan of the Wiki object,
  skipping `_index`/`_log` and deleted pages.
- `site/src/_data/sources.js` → `src:`-prefixed entries → `{blob,title,originUrl,addedBy}`.
- `site/src/_data/disputes.js` → `rpcQueryEvents` (`suix_queryEvents` for
  `<pkg>::dispute::DisputeRaised`).

Field mapping (existing output → indigo `Page`):
- `Source`: `id = blob`, `title`, `blob`, `url = aggregator blob URL`.
- `claims[]`: each marked line → `{ text, sourceIds: [marker blobs], pageBlob: latestBlob }`;
  `sourceById` resolves by blob.
- `body[]`: parse markdown into `{heading?, paragraphs:[{text, sourceIds:[markers on that line],
  bold}]}` (this is the one genuinely new transform — the Eleventy path only emitted HTML; write a
  small line/heading parser reusing `extractMarkers`).
- `versions[]`: `[latestBlob, ...history]` → `{blob, updatedAt, updatedBy}` (timestamps/author
  best-effort from `PageUpdated`/object data; acceptable to approximate).
- `links[]`: wikilink target slugs; `tags[]`: frontmatter tags; `disputes[]`: match dispute events
  by page slug → `{id, status, raisedBy, counterSource, rationale}`.
- `pos{x,y}`: assign **deterministic** layout positions in the injector (e.g. golden-angle circle by
  index) so `GraphScreen` keeps working without a force sim. (GraphScreen reads `p.pos`.)
- `diff`: do **not** precompute. Replace the static `page.diff` consumption in `PageDetailScreen`'s
  "Compare Versions" with an **on-demand client-side diff**: fetch the two selected version blobs
  from the Walrus aggregator and run the LCS from `assets/diff.js` (port to a TS util
  `app/lib/diff.ts`). This keeps time-travel real and dynamic.
- Endpoints to carry over: Sui RPC `https://fullnode.testnet.sui.io:443`, aggregator
  `https://aggregator.walrus-testnet.walrus.space/v1/blobs`.

Resilience: if `sui`/`walrus` CLI or RPC is unavailable at build, fail loudly with a clear message
(defensive JSON parsing per CLAUDE.md), and optionally fall back to a committed snapshot so the
build never hard-blocks a demo.

## Phase 3 — F11 on-chain: `cortex::attest` + upgrade + CLI (TASKS F11.1–F11.5)

- **F11.1** Locate the `UpgradeCap` minted at original publish: `sui client objects <AgentA>` filter
  `0x2::package::UpgradeCap`; record its id in `agent/.cortex/config.json` (`upgrade_cap_id`).
  Confirm gas (TASKS 8.1 showed 0 SUI/WAL — top up via faucet first). **Contingency** if UpgradeCap

- **F11.2** New module **`move/cortex/sources/attest.move`** (`cortex::attest`), per spec:

  `ProvenanceAttested` event, `attest_provenance(wiki, page, page_blob, ctx)` — **no ContributorCap**,
  `assert!(wiki::page_exists(...), E_PAGE_NOT_FOUND)`, create object, emit event,
  `transfer::public_transfer` to `ctx.sender()`; view fns `attestation_page/verifier/blob`.
- **F11.3** Move tests in `move/cortex/tests/`: attest success (object + event) and missing-page
  abort. Keep the whole package green (`sui move test --build-env testnet`).
- **F11.4** `sui client upgrade --upgrade-capability <cap> --gas-budget …`; update **Package ID** in
  `agent/.cortex/config.json` and re-run Phase-2 injector (site reads new pkg). Update the State
  section of `CLAUDE.md`.
- **F11.5** CLI **`agent/cortex_cli`** `attest <slug> [--agent a|b]`: resolve current `page_blob`

## Risks / notes

- **Package upgrade** is the main risk: needs the `UpgradeCap` (not yet recorded) + gas (was 0).
  Top up faucet and locate the cap first; fresh-publish contingency documented above. A new Package
  ID cascades to `config.json` + the site injector — re-run Phase 2 after any republish.
- **Walrus epochs**: any new `walrus store` must use `--epochs max` (rule #1). This plan mostly
  reads existing blobs; no new page blobs required unless a republish forces re-ingest.
- **SPA on Walrus Sites**: `ws-resources.json` routes is the documented pattern; HashRouter is the
  safe fallback.
- **Two-agent identity** (rule #3) is unaffected; wallet attestation is open to any address and
  strengthens, not replaces, the A/B keypair split.
